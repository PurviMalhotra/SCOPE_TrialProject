# Database Integration Contract

The database team owns PostgreSQL schema design and migrations. Event request persistence is now wired to PostgreSQL through the repository layer. Keep future database changes inside repositories and do not move SQL into routes, controllers, or services.

## Repository Boundary

All repository functions must:

- Be async.
- Return plain JavaScript objects.
- Preserve frontend-compatible field names where documented.
- Throw only infrastructure errors; business rules belong in services.
- Avoid formatting dashboard fields differently from the current contract.

## Event Request Repository

File:

```txt
src/repositories/eventRequestRepository.js
```

Required functions:

```js
findAll(filters)
findById(id)
create(payload)
update(id, payload)
remove(id)
```

Stored/returned request shape:

```js
{
  id,
  topic,
  code,
  course,
  lectureBy,
  date,
  status,
  formData,
  createdBy,
  createdAt,
  updatedAt
}
```

`formData` should preserve the original frontend payload shape. It may be stored as JSONB in PostgreSQL, or split into normalized tables as long as the repository reconstructs the same object for the service/controller layer.

## Recommended PostgreSQL Notes

These are suggestions only, not migrations:

- Use stable primary keys generated by PostgreSQL or the DB layer.
- Store `formData` as JSONB initially if fast compatibility matters.
- Index event request status and createdBy for dashboard filters.
- Index upload objectKey.
- Enforce unique users by googleId and email.
- Keep status values frontend-compatible unless the frontend is updated at the same time.

## Integration Rule

Services should not need to change when PostgreSQL is introduced. If a service change appears necessary, check whether the repository can adapt the DB shape back into the existing application contract first.
